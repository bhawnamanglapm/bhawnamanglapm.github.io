About

// the throughline
Where I started

Engineering at Ericsson on the Vietnam Mobile Operator programme, before moving into product — learning early that systems serving millions fail in ways spreadsheets never predict.

The thread

Every role — Ericsson, Bajaj Finserv, Leading Edge, now Smartsites — has dropped me into the same problem: a manual process buckling under scale, whether that's a telecom restoration script, an Excel-based payout calculation, or a 40-person design team split across time zones. My job has consistently been turning that mess into a system, usually with automation or AI doing the heavy lifting.

What I gravitate toward

Operational processes that quietly outgrew themselves — the manual handoff or spreadsheet a team routes around instead of fixing, where the real unlock is redesigning the workflow, not adding headcount.

My point of view

I believe the most under-rated product work is fixing the process nobody owns — the spreadsheet or handoff everyone's quietly working around is usually a bigger lever than the next feature.

D

AI Product Teardown — ChatGPT Memory

Feature name
Memory (persistent cross-conversation memory)
Product
ChatGPT
User problem
Users repeat context in every new conversation; the assistant "forgets" preferences, ongoing projects, and prior corrections
Business problem
Session-to-session experience feels disposable without persistence — a real retention and differentiation lever against competitors without it
Why it exists
Personalization compounds engagement — the more the assistant "knows" a user, the higher the switching cost to a competitor
What to remember vs. forget
Explicit user statements vs. inferred preferences
Guardrails
User-visible memory management (view/edit/delete), opt-out control, no silent retention of sensitive data
Evaluation challenge
No ground-truth "correct" memory — success has to be measured by correction rate and re-engagement, not accuracy
Trade-offs
Personalization vs. privacy; more useful over time vs. "creepiness" risk if memory surfaces unprompted
Failure/abuse case
Stale memory — the model over-indexes on outdated info (e.g. remembering a job the user has since left), requiring a decay/update strategy, not just accumulation
Redesign idea
Confidence-weighted memory with visible "last updated" timestamps, so users can trust and correct what's remembered in-line
Reusable insight
The general pattern for any AI feature with implicit state — the PM's real job is designing the correction and visibility mechanism, not the memory mechanism itself
